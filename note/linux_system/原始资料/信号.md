# 进程通信
## 管道通信

- **管道（Pipe）** 是最基础的**进程间通信（IPC）机制**之一，核心作用是「在两个进程间传递字节流数据」—— 本质是内核维护的一块临时内存缓冲区，数据从一个进程（写端）写入，从另一个进程（读端）读出，遵循「先进先出（FIFO）」规则。
  
---

### 一、匿名管道
- 匿名管道通过 `pipe()` 系统调用创建，内核会：
	1. 分配一块临时内存缓冲区（大小通常为 64KB，可通过 `fcntl()` 调整）；
	2. 创建两个文件描述符（`fd[0]` 和 `fd[1]`），分别对应「读端」和「写端」；
	    - `fd[0]`：仅用于读取数据（关闭写权限）；
	    - `fd[1]`：仅用于写入数据（关闭读权限）；
	3. 进程通过读写这两个文件描述符，间接操作内核缓冲区（数据不经过磁盘，纯内存操作，效率高）。
	核心数据流：`写进程 → fd[1] → 内核缓冲区 → fd[0] → 读进程`
- `#include <unistd.h> int pipe(int pipefd[2]);`
  功能：创建一个用于进程通信的单向数据管道
  参数：输出型参数，数组大小为 2，存储管道的两个文件描述符，0只读，1只写
  返回值：成功返回0，失败返回-1，设置errno
- 匿名管道只能用于亲缘进程通信，核心流程是「先创建管道，再 `fork()` 子进程」，因为 `fork()` 会复制父进程的文件描述符，子进程能继承管道的 `fd[0]` 和 `fd[1]`，进程结束后管道消失。
- 匿名管道是进程间半双工通信。写端阻塞条件：当管道满时（例如 64KB），`write()` 进入阻塞，直到：有进程执行 read或管道被关闭（收到 SIGPIPE），读端阻塞条件：当管道空时，如果 _有写端仍打开_ → read 阻塞等待，如果 _所有写端关闭_ → read 返回 0（EOF）。

---

### 二、命名管道
- 命名管道是一种基于 **文件系统路径** 的半双工 IPC（进程间通信）方式。有文件系统节点，不要求亲缘关系，适用于任何进程，进程结束后`FIFO`文件留在系统中，文件无任何内容。命名管道只是一种“入口文件”，**不存储数据**，数据实际仍在内核管道缓冲区中。
- `#include <sys/types.h>,#include <sys/stat.h>,mkfifo("myfifo", 0666);`
- `open`以只读/写方式打开管道文件时会默认阻塞，直到一个对应端打开，在打开设备文件（终端、串口等）时依设备驱动发生阻塞。

## System V IPC

### 0. 概述与核心概念

System V IPC 包括三种机制：消息队列、共享内存、信号量。

它们都使用一个共同的标识符 Key (key_t) 来在内核中定位资源。

#### 关键标识符 Key 生成：`ftok`

所有 IPC 操作的第一步。将文件路径和项目 ID 转换为唯一的 System V IPC 键值。

- **头文件**: `<sys/types.h>`, `<sys/ipc.h>`
    
- **原型**

   ```
    key_t ftok(const char *pathname, int proj_id);
    ```

- **参数**:
    
    - `pathname`: 一个**必须存在**且可访问的文件路径。
        
    - `proj_id`: 项目 ID，低 8 位有效（通常用字符常量，如 `'A'`）。
        
- **返回值**:
    
    - 成功: 返回生成的 `key_t` 值。
        
    - 失败: 返回 `-1`，并设置 `errno`。
        

---

### 1. 消息队列 (Message Queues)

**特点**: 数据以“块”为单位，有类型，自带同步机制。

- **头文件**: `<sys/types.h>`, `<sys/ipc.h>`, `<sys/msg.h>`
    

#### 1.1 数据结构定义 (必须遵守)

虽然函数参数是 `void *`，但发送的数据**必须**符合以下内存布局：

```
struct msgbuf {
    long mtype;       // 【必需】消息类型，必须 > 0
    char mtext[1];    // 【可选】消息正文，长度由用户定义，可以是结构体/数组
};
```

#### 1.2 创建/获取：`msgget`

```
int msgget(key_t key, int msgflg);
```

- **msgflg**:
    
    - `IPC_CREAT`: 不存在则创建，存在则打开。
        
    - `IPC_EXCL`: 与 `IPC_CREAT` 连用，若已存在则报错。
        
    - 权限位 (如 `0666`)。
        
- **返回值**: 成功返回 `msqid` (非负整数)，失败返回 `-1`。
    

#### 1.3 发送消息：`msgsnd`

```
int msgsnd(int msqid, const void *msgp, size_t msgsz, int msgflg);
```

- **msgp**: 指向消息结构体的指针。
    
- **msgsz**: **消息正文的大小**（字节数）。
    
    - ⚠️ **注意**: `msgsz = sizeof(struct msgbuf) - sizeof(long)`。不要包含 `mtype` 的大小。
        
- **msgflg**:
    
    - `0`: 队列满时**阻塞**。
        
    - `IPC_NOWAIT`: 队列满时返回 `-1` (errno: `EAGAIN`)。
        

#### 1.4 接收消息：`msgrcv`

```
ssize_t msgrcv(int msqid, void *msgp, size_t msgsz, long msgtyp, int msgflg);
```

- **msgtyp**:
    
    - `0`: 读取队列中**第一条**消息（FIFO）。
        
    - `> 0`: 读取类型**等于** `msgtyp` 的第一条消息。
        
    - `< 0`: 读取类型**小于等于** `abs(msgtyp)` 的消息中类型值最小的一条（优先级读取）。
        
- **msgflg**:
    
    - `IPC_NOWAIT`: 无消息时不阻塞。
        
    - `MSG_NOERROR`: 若消息长度超过 `msgsz`，则截断消息且不报错。
        

#### 1.5 控制/删除：`msgctl`

```
int msgctl(int msqid, int cmd, struct msqid_ds *buf);
```

- **cmd**:
    
    - `IPC_RMID`: **立即删除**消息队列（最常用）。
        
    - `IPC_STAT`: 获取队列状态。

### 2. 共享内存 (Shared Memory)

**特点**: 最快的 IPC，无内核拷贝，需要自行处理同步（通常搭配信号量）。

- **头文件**: `<sys/ipc.h>`, `<sys/shm.h>`
    

#### 2.1 创建/获取：`shmget`

```
int shmget(key_t key, size_t size, int shmflg);
```

- **size**: 共享内存段的大小（字节）。如果是获取已存在的，设为 `0` 即可。
    
    - 建议是页大小 (4096) 的倍数。
        
- **返回值**: 成功返回 `shmid`。
    

#### 2.2 挂载 (映射)：`shmat`

将共享内存段连接到当前进程的地址空间。

```
void *shmat(int shmid, const void *shmaddr, int shmflg);
```

- **shmaddr**: 通常设为 `NULL`，让系统自动选择合适的地址。
    
- **shmflg**: `0` (读写)，`SHM_RDONLY` (只读)。
    
- **返回值**: 成功返回映射好的内存指针 `void *` (类似 `malloc` 的返回)，失败返回 `(void *)-1`。
    

#### 2.3 分离 (解除映射)：`shmdt`

断开进程与共享内存的连接（**不是删除**共享内存）。

```
int shmdt(const void *shmaddr);
```

- **shmaddr**: `shmat` 返回的指针。
    

#### 2.4 控制/删除：`shmctl`

```
int shmctl(int shmid, int cmd, struct shmid_ds *buf);
```

- **cmd**: `IPC_RMID` (标记删除)。
    
    - ⚠️ **注意**: 共享内存会有引用计数。执行 `IPC_RMID` 后，只有当所有进程都执行了 `shmdt`，内存才会真正释放。
        

---

### 3. 信号量 (Semaphores)

**特点**: 用于进程同步与互斥的计数器集合。

- **头文件**: `<sys/types.h>`, `<sys/ipc.h>`, `<sys/sem.h>`
    

#### 3.1 创建/获取：`semget`

```
int semget(key_t key, int nsems, int semflg);
```

- **nsems**: 信号量集合中信号量的数量（通常为 1）。
    
- **返回值**: 成功返回 `semid`。
    

#### 3.2 操作 (P/V操作)：`semop`

原子性地对信号量进行加减操作。

```
int semop(int semid, struct sembuf *sops, size_t nsops);
```

- **sops**: 指向操作结构体数组的指针。

  ```
    struct sembuf {
        unsigned short sem_num;  // 信号量索引 (从 0 开始)
        short          sem_op;   // 操作值: -1 (P/Wait), +1 (V/Signal), 0 (等待直到为0)
        short          sem_flg;  // 标志: SEM_UNDO (进程崩溃时自动撤销操作，防死锁)
    };
    ```

- **nsops**: 结构体数组的长度（即一次执行几个操作）。

#### 3.3 初始化/控制：`semctl`

```
int semctl(int semid, int semnum, int cmd, ...);
```

- **参数**: 第四个参数是**联合体 (Union)**，必须由用户自己定义（有些系统头文件不包含它）：

```
    union semun {
        int              val;    // 用于 SETVAL 的值
        struct semid_ds *buf;    // 用于 IPC_STAT, IPC_SET
        unsigned short  *array;  // 用于 GETALL, SETALL
    };
``` 
- **cmd**:
	
    - `SETVAL`: 设置信号量的初值 (如设为 1 实现互斥锁)。
        
    - `IPC_RMID`: 删除信号量集合。

# 信号
- 信号是软件模拟的软中断，可以使用信号进行进程间的异步通讯。`Linux`系统提供了64种中断信号，信号是发送给进程的信号，每个进程都有对信号的默认处理函数，默认处理为终止当前进程。进程对信号的处理函数可以被子进程继承。 
## 一、信号处理函数
- `#include <signal.h>,typedef void (*sighandler_t)(int);sighandler_t signal(int signum, sighandler_t handler);`
  **功能**：为给定的信号设置处理方式
  **参数**：
	  signum：信号编号
	  handler：
		  SIG_IGN：忽略
		  SIG_DFL：默认的处理函数
		  自定义函数
  **返回值**：错误返回SIG_ERR，成功返回原来的值

---

## 二、信号的产生
  1. 硬件：`ctrl+c`,`ctrl+d`,`ctrl+\`
  2. `linux`命令：`kill -signum pid`
  3. 使用函数产生信号：`kill`,`alarm`,`raise`
	- `#include <unistd.h>,unsigned int alarm(unsigned int seconds);`
		  **功能**：给当前进程设置一个 “定时闹钟”—— 指定 `seconds` 秒后，内核会向进程发送 `SIGALRM` 信号（默认行为是终止进程）。同时明确了两个关键规则：“秒数为 0 取消未触发的闹钟” 和 “新调用会覆盖之前的闹钟”。进程退出后闹钟消失，子进程不继承闹钟。
		  **参数**：`seconds`闹钟倒计时时长，单位为秒，为0时取消未触发的闹钟
		  **返回值**：成功设置新闹钟时，返回「上一个闹钟剩余的秒数」（若之前无闹钟，返回 0）， 失败返回 -1
	- `#include <unistd.h>,int pause(void);`
		  **功能**：**让当前进程（或线程）“挂起休眠”**，直到收到一个 “能唤醒它的信号”。 要么是终止进程的信号（如 `SIGINT`），要么是有自定义处理函数的信号（如注册过 `handler` 的 `SIGALRM`）
		  **返回值**：成功唤醒（信号被捕获且 `handler` 返回）：返回 `-1`，并设置 `errno = EINTR`，- 其他情况：要么进程被终止（不返回），要么一直休眠（永不返回）。

## 三、信号阻塞与未决信号
- 信号的阻塞和未决信号：开发人员可依据实际对特定信号进阻塞，信号抵达后不对信号进行处理，解除对信号的阻塞后再对信号进行相应处理。系统提供了一系列函数用于相应设置信号集类型`sigset_t`本质是一个「位图（bitmask）」，每个比特位对应一个信号。1表示信号被包含在集合内，0表示被排除。
	  1. `int sigemptyset(sigset_t *set);`全初始化为0
	  2. `int sigfillset(sigset_t *set);`全初始化为1
	  3. `int sigaddset(sigset_t *set, int signum);`添加信号
	  4. `int sigdelset(sigset_t *set, int signum);`删除信号
	  5. `int sigismember(const sigset_t *set, int signum);`检查信号是否在集合中
	  6. `int sigprocmask(int how, const sigset_t *_Nullable restrict set, sigset_t *_Nullable restrict oldset);`获取或修改当前线程的信号集
- `#include <signal.h>,int sigpending(sigset_t *set);`
	  **功能**：查询当前进程 / 线程的 “未决信号集
	  **参数**：输出型参数：用于存储内核返回的「未决信号集」，调用成功后，`set` 中比特位为 `1` 的信号就是当前未决的信号
	  **返回值**：成功返回 `0`；失败返回 `-1`（设置 `errno`，常见 `EINVAL`：`set` 是无效指针）
- **不可靠信号（Unreliable Signal）** 指的是「1-31 号普通信号」（对应 `kill -l` 输出中 1-31 号信号，问题是「不支持排队、可能丢失，且信号处理期间可能被同类型信号打断」，**可靠信号**指34 号及以后的实时信号，支持排队、不丢失、默认阻塞同类型信号。
- **可重入函数**：只使用函数私有空间，不使用进程全局空间的函数。可重入函数是指那些“可以被安全地中断，并在中断期间再次被调用”的函数，主要有信号处理和多线程两种使用场景。不可重入函数通常使用了`static`变量或`global`变量；使用了`malloc`或`free`；使用了标准`IO`库，`stdio`

## 四、进程与会话

- `session`：一组共享同一个 **SID (Session ID)** 的进程组。会话通常有一个**会话 Leader**，通常是启动会话的进程（例如登录 Shell 或终端模拟器）。每个会话可以关联一个控制终端（Control Terminal）。一个会话只能有一个**前台进程组**，可以有多个后台进程组。当会话 Leader 退出或终端关闭时，内核会向会话中的所有进程发送 `SIGHUP` 信号。会话 (Session) 是内核用来组织进程组的抽象概念，而**控制终端**只是**可能**与这个会话关联的一个I/O设备。所以，一个进程使用 `setsid()` 创建新会话的主要目的，就是为了保证该会话**没有控制终端**，从而免疫于终端关闭等事件，实现守护进程的功能。

- 守护进程：**守护进程**（Daemon Process）是 Unix/Linux 系统中一种**特殊的后台进程**。它独立于控制终端并周期性地或在特定事件发生时执行某些任务，为整个系统提供核心服务。一个标准的守护进程必须满足以下几个关键要求，使其能够独立于任何用户登录会话而稳定运行：
	1. **脱离控制终端 (No Controlling TTY):** 守护进程不与任何终端关联，因此不会受到终端发出的信号（如 `SIGINT`, `SIGQUIT`）或关闭终端时发送的 `SIGHUP` 信号的影响。 
	2. **后台运行：** 它通常在系统启动时被初始化，并在后台持续运行，直到系统关闭。
	3. **新的会话领导者：** 守护进程会创建一个新的 Session（无控制终端），并成为该会话的 Leader，彻底切断与父进程（通常是 Shell）和原有 Session 的联系。
