# 网络编程与服务器总览

## 1. 这组笔记解决什么问题

这组笔记不是 socket API 清单，也不是代码仓库说明，而是记录服务器学习中不断演进的几个核心问题：

```text
一个连接会在哪里阻塞？
fd 由谁拥有，什么时候关闭？
I/O 等待和业务处理是否混在一起？
多进程、多线程、线程池、select、poll、epoll分别解决什么问题？
每次模型升级引入了什么新的工程风险？
```

## 2. 当前最重要的主线

```text
单进程阻塞：理解 TCP server 最小闭环
多进程：理解 fork、fd 表副本和僵尸进程
多线程：理解共享 fd 表和并发风险
线程池：理解任务队列和 worker 复用
select：理解一个线程管理多个 fd 的就绪事件
poll：理解 pollfd 数组和 revents
epoll：理解事件通知机制向 Reactor 方向演进
```

## 3. 阶段关系

| 阶段 | 解决的问题 | 引入的新问题 |
|---|---|---|
| 单进程阻塞 | 打通 TCP server 最小流程 | 一次只能处理一个客户端 |
| 多进程 | 父进程继续 accept，子进程处理连接 | 进程开销、僵尸进程、fd 引用关系 |
| 多线程 | 并发成本低于多进程 | 共享 fd 表、线程安全、日志交叉 |
| 线程池 | 控制线程数量，复用 worker | 慢连接占用 worker，任务边界不清 |
| select | 一个 I/O 线程监听多个 fd | fd_set、max_fd、遍历成本、生命周期维护 |
| poll | 去掉 fd_set/max_fd 限制 | 仍要遍历 pollfd 数组 |
| epoll | 内核维护监听集合，返回就绪事件 | 事件模式、连接对象、缓冲区设计更重要 |

## 4. 当前阶段重点

当前应优先巩固：

```text
fd 生命周期
I/O 多路复用事件循环
I/O 线程与 ThreadPool 边界
select -> poll -> epoll 的相同点和差异
Connection / read_buffer / write_buffer 的引入时机
```

HTTP 暂时作为后续路线，不在当前 I/O 多路复用笔记里展开。
