# 阻塞、多进程、多线程与线程池服务器

> 本篇目标：整理从最小 TCP Echo Server 到线程池服务器的演进。  
> 关注点：阻塞点、执行流、fd 生命周期、进程/线程差异、线程池任务模型。  
> 暂不涉及 select / poll / epoll，也不涉及 HTTP。

---

# 1. 总览：这一阶段在解决什么

最初的阻塞 TCP 服务器只能服务一个客户端：

```text
socket()
bind()
listen()
accept()
read()
write()
close()
```

它的问题不是 socket 流程不完整，而是：

```text
accept 会阻塞等待新连接；
read 会阻塞等待当前客户端数据；
只要一个连接卡住，服务器就无法处理其他连接。
```

于是服务器模型开始演进：

```text
单进程阻塞
    -> 多进程
    -> 多线程
    -> 线程池
```

每一步都在解决上一层的问题，同时引入新的问题。

| 阶段 | 解决的问题 | 新问题 |
|---|---|---|
| 单进程阻塞 | 打通 TCP server 最小闭环 | 一次只能处理一个客户端 |
| 多进程 | 父进程继续 accept，子进程处理连接 | 进程开销大，僵尸进程，需要理解 fd 表副本 |
| 多线程 | 创建成本低于进程，共享地址空间 | 共享 fd 表，并发安全，日志交叉，生命周期混乱 |
| 线程池 | 控制线程数量，复用 worker | 慢连接长期占用 worker，任务边界需要清晰 |

---

# 2. 单进程阻塞 TCP Echo Server

## 2.1 模型

```text
main thread:
    socket
    bind
    listen
    while true:
        client_fd = accept(listen_fd)
        while true:
            n = read(client_fd)
            if n <= 0: break
            write(client_fd)
        close(client_fd)
```

## 2.2 必须掌握

### socket 生命周期

```text
socket()     创建监听 socket
setsockopt() 设置地址复用等选项
bind()       绑定 IP/port
listen()     进入监听状态
accept()     从已完成连接队列中取出一个连接
read/write   与客户端通信
close()      关闭 fd
```

### 两个核心阻塞点

```text
accept 阻塞：没有新连接时，服务器停在 accept。
read 阻塞：当前客户端不发数据时，服务器停在 read。
```

## 2.3 工程意义

单进程阻塞版本虽然简单，但它是所有后续模型的基础。后续无论是多进程、多线程、select、poll、epoll，本质上仍然绕不开：

```text
listen_fd 负责接收新连接；
client_fd 负责和某个客户端通信；
read 返回 0 表示对端关闭；
read/write 都可能失败；
close 必须和 fd 生命周期对应。
```

## 2.4 常见误区

```text
误区 1：以为 read 返回 0 是读到字符 '0'。
修正：read/recv 返回 0 表示对端有序关闭。

误区 2：以为 TCP 一次 write 对应对端一次 read。
修正：TCP 是字节流，没有应用层消息边界。

误区 3：只记 API 顺序，不理解阻塞点。
修正：服务器模型演进的核心就是处理阻塞点。
```

# 3. 多进程服务器

## 3.1 模型

```text
父进程：
    while true:
        client_fd = accept(listen_fd)
        pid = fork()
        if pid == 0:
            close(listen_fd)
            handle_client(client_fd)
            close(client_fd)
            exit(0)
        else:
            close(client_fd)
            continue accept
```

## 3.2 fork 后 fd 表的关键理解

`fork()` 后父子进程有各自的 fd 表副本：

```text
父进程 close(client_fd) 不会直接影响子进程继续使用这个连接。
子进程 close(listen_fd) 不会影响父进程继续 accept。
```

更准确地说，父子进程的 fd 数字相同，但它们是两个进程 fd 表中的条目，引用同一个内核打开文件/socket 对象。只有所有引用都关闭后，内核对象才真正释放。

## 3.3 僵尸进程

子进程退出后，父进程需要回收它的退出状态，否则会产生僵尸进程。

常见处理：

```cpp
signal(SIGCHLD, SIG_IGN);
```

更严谨做法：

```text
sigaction + waitpid(-1, &status, WNOHANG)
```

## 3.4 工程意义

多进程模型的优点：

```text
进程之间隔离强；
一个子进程崩溃不一定影响父进程；
模型直观，适合理解 fork 与 fd 继承。
```

缺点：

```text
创建进程成本高；
上下文切换开销大；
进程间共享状态麻烦；
需要处理 SIGCHLD 和僵尸进程。
```

## 3.5 常见误区

```text
误区 1：父进程 close(client_fd) 会导致子进程不能读写。
修正：fork 后父子进程有独立 fd 表条目。

误区 2：子进程处理完后不 exit。
后果：子进程可能继续执行父进程 accept 逻辑，造成混乱。

误区 3：只 fork 不 wait。
后果：子进程退出后变成僵尸进程。
```

# 4. 多线程服务器

## 4.1 模型

```text
主线程：
    while true:
        client_fd = accept(listen_fd)
        std::thread(handle_client, client_fd).detach()

工作线程：
    handle_client(client_fd)
    close(client_fd)
```

## 4.2 与多进程的核心区别

```text
多进程：父子进程有各自 fd 表。
多线程：同一进程内所有线程共享同一张 fd 表。
```

因此：

```text
主线程 accept 后不能随便 close(client_fd)。
如果主线程 close 了，工作线程使用同一个 fd 数字时可能得到 EBADF。
```

## 4.3 参数传递风险

错误写法倾向：

```cpp
std::thread t(handle_client, std::ref(client_fd));
```

问题：主线程循环中的 `client_fd` 变量会变化，多个线程可能看到同一个变量的新值。

推荐：

```cpp
std::thread t(handle_client, client_fd);
```

把 fd 数值按值传递给线程。

## 4.4 日志并发问题

多个线程同时输出：

```cpp
std::cout << "client connected" << std::endl;
```

可能产生交叉输出。

教学阶段可以：

```cpp
std::mutex cout_mutex;
std::lock_guard<std::mutex> lock(cout_mutex);
```

工程上更推荐：

```text
日志队列 + 单独日志线程
```

## 4.5 工程意义

多线程模型让你开始面对 C++ 并发和 Linux fd 生命周期的结合：

```text
线程共享进程资源；
fd 是进程级资源；
close 是全进程可见的；
锁保护的是共享数据，不是 fd 本身；
线程函数的参数和生命周期必须清晰。
```

# 5. 线程池阻塞服务器

## 5.1 为什么需要线程池

每个连接创建一个线程的问题：

```text
连接数上升时线程数无限增长；
线程创建/销毁开销大；
上下文切换变多；
系统资源不可控。
```

线程池解决：

```text
固定 worker 数量；
复用线程；
通过任务队列提交连接处理任务。
```

## 5.2 基本模型

```text
main thread:
    创建 TcpServer
    创建 ThreadPool
    while true:
        client_fd = accept()
        pool.submit([client_fd] {
            handle_client(client_fd);
        })

worker thread:
    从任务队列取任务
    执行 handle_client(client_fd)
```

## 5.3 ThreadPool 核心组成

```cpp
std::vector<std::thread> workers_;
std::queue<std::function<void()>> tasks_;
std::mutex mutex_;
std::condition_variable condition_;
bool stop_;
```

职责：

```text
创建 worker；
维护任务队列；
submit() 添加任务；
worker_loop() 等待并执行任务；
析构时通知停止并 join。
```

## 5.4 关键规则：锁内取任务，锁外执行任务

正确结构：

```cpp
while (true) {
    std::function<void()> task;

    {
        std::unique_lock<std::mutex> lock(mutex_);
        condition_.wait(lock, [this] {
            return stop_ || !tasks_.empty();
        });

        if (stop_ && tasks_.empty()) {
            return;
        }

        task = std::move(tasks_.front());
        tasks_.pop();
    }

    task();
}
```

重点：

```text
锁只保护任务队列。
任务可能执行很久，甚至阻塞 read。
不能持有队列锁执行 task。
```

## 5.5 线程池阻塞模型的局限

如果任务是：

```cpp
void handle_client(int client_fd) {
    while (true) {
        read(client_fd, ...);
        write(client_fd, ...);
    }
    close(client_fd);
}
```

那么一个慢客户端就会长期占用一个 worker。

当连接数超过 worker 数量时：

```text
worker 全部阻塞在 read；
新连接任务排队；
线程池虽然控制了线程数，但并没有解决 I/O 阻塞占用 worker 的问题。
```

这正是后来进入 `select / poll / epoll` 的原因。

# 6. 这一篇的核心总结

```text
单进程阻塞：理解最小 TCP 流程。
多进程：理解 fork、fd 表副本、僵尸进程。
多线程：理解线程共享 fd 表、并发输出、参数传递。
线程池：理解任务队列、条件变量、锁内取任务锁外执行。
```

最重要的阶段结论：

```text
线程池能控制线程数量，但如果任务内部仍然阻塞 read，worker 仍然会被慢连接占用。
```

所以后续要引入 I/O 多路复用：

```text
不要让 worker 阻塞等待某个客户端数据；
让一个 I/O 线程统一等待多个 fd 的就绪事件。
```
