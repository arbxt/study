# 07 fd 生命周期与连接管理

## 1. fd 的本质

`fd` 是当前进程 fd 表中的整数索引。它不是文件本身，也不是 socket 本身。

```text
fd -> 内核中的打开文件对象 / socket 对象
```

socket 场景中：

```text
fd -> socket 对象 -> TCP 状态 + 接收缓冲区 + 发送缓冲区
```

## 2. 长连接中的 client_fd

重要结论：

```text
client_fd 可以长期打开。
不是每次处理完业务就 close。
```

“打开 fd”表示：

```text
进程仍然持有这个 fd，还没有 close。
```

不是：

```text
正在 read/write。
```

## 3. 长连接流程

```text
accept 得到 client_fd
FD_SET 加入监听
select 等待可读
recv 一次
提交业务
保留 fd
等待下一次可读
直到 recv 返回 0 或不可恢复错误
remove_client
```

## 4. 什么时候关闭 fd

应该关闭：

```text
recv 返回 0。
recv 返回不可恢复错误。
协议规定短连接处理完即关闭。
超时。
服务器主动关闭。
程序退出。
```

不应该因为：

```text
这次业务处理完了。
```

就立刻关闭长连接 fd。

## 5. remove_client 的完整职责

```cpp
void Selector::remove_client(int client_fd)
{
    close(client_fd);
    FD_CLR(client_fd, &master_set_);

    client_fds_.erase(
        std::remove(client_fds_.begin(), client_fds_.end(), client_fd),
        client_fds_.end()
    );

    if (client_fd == max_fd_) {
        rebuild_max_fd();
    }
}
```

必须成组维护：

```text
close
FD_CLR
erase
rebuild_max_fd if needed
```

## 6. closed_fds_ 模式

遍历 `client_fds_` 时不要直接删除元素，推荐：

```text
本轮先把需要关闭的 fd 记录到 closed_fds_。
遍历结束后统一 remove_client。
最后 clear closed_fds_。
```

如果忘记 clear：

```text
下一轮可能重复 remove 同一个 fd。
fd 数字被系统复用后，甚至可能误关新连接。
```

## 7. rebuild_max_fd

正确思路：

```text
先重置为 listen_fd。
再遍历 client_fds_ 找最大值。
```

不能在旧 `max_fd_` 基础上继续比较。
