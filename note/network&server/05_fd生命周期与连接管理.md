# fd 生命周期与连接管理

## 1. 为什么 fd 生命周期是服务器核心

服务器最容易出错的不是 `socket()` 怎么写，而是：

```text
fd 谁拥有？
谁能 read？
谁能 write？
谁能 close？
close 后谁负责从监听集合移除？
fd 被复用后会不会误操作新连接？
```

## 2. threadpool 版本中的 fd 生命周期

```text
main accept 得到 client_fd
main 把 client_fd 值捕获提交给 worker
worker handle(client_fd)
worker read/write
worker close(client_fd)
```

所有权判断：

```text
client_fd 从 main 转移给 worker。
main 不再操作该 client_fd。
worker 是生命周期拥有者。
```

因此：

```text
worker close 是合理的。
```

## 3. select_pool 版本中的 fd 生命周期

```text
Selector accept 得到 client_fd
Selector FD_SET 加入 master_set_
Selector push_back 到 client_fds_
Selector select 等待可读
Selector recv 一次
Selector 提交 data 到 worker
Selector 在 recv==0 或错误时 remove_client
```

所有权判断：

```text
client_fd 生命周期归 Selector 管理。
worker 当前只是临时借用 client_fd 写响应。
```

因此：

```text
worker 不应该 close client_fd。
worker 不应该 read client_fd。
```

## 4. remove_client 的完整职责

当前项目写法：

```cpp
void Selector::remove_client(int client_fd) {
    close(client_fd);
    FD_CLR(client_fd, &master_set_);
    client_fds_.erase(
        std::remove(client_fds_.begin(), client_fds_.end(), client_fd),
        client_fds_.end()
    );

    if (client_fd == max_fd_) {
        rebuild_max_fd();
    }
}
```

这组操作必须成组出现：

```text
close
FD_CLR
从 client_fds_ 删除
必要时 rebuild_max_fd
```

只 `close(fd)` 是不完整的。

## 5. closed_fds_ 的作用

```text
遍历 client_fds_ 时不要立刻删除元素。
先把待关闭 fd 放入 closed_fds_。
遍历结束后统一 remove_client。
最后 clear。
```

模式：

```cpp
for (int fd : client_fds_) {
    if (FD_ISSET(fd, &read_set_)) {
        handle_client_event(fd);
    }
}

for (int fd : closed_fds_) {
    remove_client(fd);
}
closed_fds_.clear();
```

如果忘记 clear：

```text
下一轮可能重复 remove 旧 fd。
如果 fd 数字被系统复用，可能误关新连接。
```

## 6. fd 复用风险

Linux 会复用较小的可用 fd。

例如：

```text
旧连接 fd = 5
Selector close(5)
新连接 accept 后可能又得到 fd = 5
旧 worker 任务如果还持有 5，就可能写到新连接
```

这就是 worker 直接持有裸 fd 的根本风险。

## 7. 后续改进方向

### 方案一：短期学习版

```text
保留 worker write fd。
明确记录风险。
控制测试规模。
```

### 方案二：连接对象 + generation

```cpp
struct ConnectionId {
    int fd;
    uint64_t generation;
};
```

每次 accept 增加 generation，worker 返回时检查是否仍然是同一连接。

### 方案三：标准 Reactor

```text
I/O 线程负责 read/write/close。
worker 只处理业务，生成 response。
response 放回连接写缓冲。
I/O 线程监听可写事件并发送。
```

这是后续 epoll + HTTP Server 的目标。
