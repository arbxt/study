# 阻塞服务器与线程池模型

## 1. 单进程阻塞模型

```text
accept 阻塞等待新连接
read 阻塞等待客户端数据
write 写回数据
close 关闭连接
```

问题：

```text
一个客户端卡在 read，服务器就无法处理其他客户端。
```

## 2. 每连接一线程模型

```text
main:
    while true:
        client_fd = accept()
        std::thread(handle, client_fd).detach()

worker:
    while read(client_fd):
        write(client_fd)
    close(client_fd)
```

优点：

```text
能同时处理多个客户端。
代码结构容易理解。
```

问题：

```text
每个连接创建一个线程。
连接数上升后线程数不可控。
上下文切换和内存压力变大。
```

## 3. 线程池阻塞模型

项目目录：

```text
project/server/threadpool
```

核心代码模型：

```cpp
while (true) {
    int client_fd = server.accept_client();

    if (client_fd == -1) {
        continue;
    }

    pool.submit([client_fd]() {
        handle(client_fd);
    });
}
```

这里的关键是：

```text
client_fd 必须值捕获。
不能引用捕获。
```

错误写法：

```cpp
pool.submit([&]() {
    handle(client_fd);
});
```

原因：

```text
client_fd 是循环局部变量。
引用捕获会导致多个任务引用同一个变量。
后续循环改变 client_fd 后，旧任务可能拿到错误 fd。
```

## 4. handle 的职责

在 `threadpool` 版本中，`handle(client_fd)` 的职责是完整的：

```text
read
write
判断断开
close
```

因为该模型中：

```text
main accept 后把 client_fd 交给 worker。
worker 拥有这个 client_fd 的生命周期。
```

因此 worker 最后 `close(client_fd)` 是合理的。

## 5. 模型局限

线程池阻塞模型的根本问题：

```text
一个连接对应一个长期任务。
一个慢客户端可能长期占用一个 worker。
```

如果线程池只有 4 个 worker，4 个客户端连接后不发数据，则 4 个 worker 都会阻塞在 `read`。

这时：

```text
main 仍然可以 accept 新连接。
但新连接提交的任务没有 worker 执行。
```

这就是后续引入 `select` 的原因。

## 6. 与 select + threadpool 的关键区别

| 对比项 | 线程池阻塞模型 | select + threadpool 模型 |
|---|---|---|
| 谁 accept | main | Selector |
| 谁 read/recv | worker | Selector |
| 谁 close client_fd | worker | Selector |
| worker 任务粒度 | 一个连接 | 一次已读数据/一次业务 |
| 是否长期阻塞 worker | 是 | 理论上不应该 |

这个区别非常重要。不要把旧版 `handle` 直接提交给 `select_pool` 的线程池。
