# 测试、压测与调试

## 1. nc 基础测试

启动服务器：

```bash
./bin/server
```

连接：

```bash
nc 127.0.0.1 8080
```

输入：

```text
hello
```

预期：

```text
服务端打印收到数据。
客户端收到 echo。
```

## 2. 测试客户端关闭

推荐：

```bash
nc -N 127.0.0.1 8080
```

然后按：

```text
Ctrl+D
```

服务端应检测到：

```text
recv/read 返回 0
closed_fds_ 记录 fd
remove_client 关闭连接
```

注意：普通 `nc` 的 Ctrl+D 不一定立刻关闭 TCP socket，可能导致服务端没有立即收到 FIN。

## 3. 多客户端测试

开多个终端：

```bash
nc 127.0.0.1 8080
```

观察：

```text
多个 client_fd 是否加入 select 集合。
每个客户端是否能 echo。
某个客户端断开是否影响其他客户端。
```

## 4. 压测工具

项目中有：

```text
project/server/utils/tcp_bench_short.cpp
```

功能：

```text
创建多个线程。
每个线程发起多次短连接请求。
统计成功数、失败数、QPS、平均延迟、P95、P99。
```

示例：

```bash
g++ -std=c++17 -O2 -pthread tcp_bench_short.cpp -o tcp_bench_short
./tcp_bench_short 127.0.0.1 8080 100 100
```

观察指标：

```text
总请求数
成功数
失败数
总耗时
QPS
平均延迟
P95 延迟
P99 延迟
```

## 5. 观察线程数

```bash
ps -T -p <PID>
ls /proc/<PID>/task | wc -l
```

用于对比：

```text
每连接一线程
线程池
select + threadpool
```

pidstat -w -p <PID> 1
```

关注：

cswch/s
nvcswch/s
```

## 7. 常见故障定位

### 7.1 Bad file descriptor

可能原因：

```text
fd 被提前 close。
worker close 了 Selector 仍在监听的 fd。
closed_fds_ 没有 clear，重复关闭。
fd 被复用后误操作。
```

### 7.2 select 返回 EBADF

可能原因：

```text
master_set_ 中残留已经 close 的 fd。
close 后忘记 FD_CLR。
client_fds_ 与 master_set_ 不一致。
```

### 7.3 客户端无 echo

可能原因：

```text
worker 没有执行任务。
线程池被阻塞任务占满。
write_all 失败。
client_fd 已被关闭。
```

## 8. 推荐测试清单

```text
1. 单客户端 echo
2. 多客户端 echo
3. 客户端主动断开
4. 多客户端中一个断开，其他保持可用
5. 大量短连接压测
6. 慢客户端连接后不发数据
7. 重复启动服务器检查 SO_REUSEADDR
8. 使用 strace 观察系统调用
```
