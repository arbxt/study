# 后续演进路线

> 本文只作为路线索引，不展开 HTTP 具体协议实现。当前优先级仍是 I/O 多路复用、fd 生命周期和连接对象。

# 11 poll / epoll / HTTP 演进路线

## 1. 当前不建议直接跳 epoll

原因：

```text
poll/epoll 只是事件通知机制变化。
fd 生命周期、资源归属、任务边界问题不会自动消失。
```

如果 select + pool 中还没搞清楚：

```text
client_fd 谁拥有？
什么时候 close？
worker 能不能直接 write？
fd 关闭后任务队列中旧 fd 怎么办？
同一个 fd 能不能重复提交给线程池？
```

那么跳到 epoll 只会把问题隐藏得更深。

## 2. 推荐顺序

```text
1. 修正当前 select + pool 的 fd 生命周期问题。
2. 做多客户端和断开测试。
3. 改 poll。
4. 写 epoll 单线程 echo。
5. 再做 epoll + pool。
6. 演进 HTTP server。
```

## 3. poll 阶段

核心变化：

```text
用 vector<pollfd> 替换 fd_set / max_fd / rebuild_max_fd。
```

不变的问题：

```text
client_fd 生命周期。
连接集合维护。
recv 返回值处理。
worker 与 I/O 的边界。
```

## 4. epoll 单线程阶段

重点理解：

```text
epoll_create / epoll_ctl / epoll_wait
EPOLLIN / EPOLLOUT
水平触发 / 边缘触发
非阻塞 fd
```

不要一上来就写复杂 Reactor。先写单线程 echo。

## 5. epoll + pool 阶段

目标：

```text
I/O 尽量回到 epoll 线程。
Pool 只做业务。
业务生成 response 后交回连接写缓冲。
```

## 6. HTTP Server 阶段

新增主题：

```text
请求行解析
Header 解析
Body 处理
Content-Length
Connection: keep-alive
状态码
静态文件
简单 JSON API
日志
配置
定时器
```

## 7. 长期目标

从 echo server 演进为：

```text
带配置、日志、连接管理、定时器、HTTP 解析、简单 API、数据库或缓存的 C++ 后端服务。
```


## 当前建议

```text
1. 先确认阻塞/多进程/多线程/线程池模型边界。
2. 再稳定 select/poll/epoll 的事件循环和 fd 生命周期。
3. 引入 Connection 对象，保存 fd、读缓冲、写缓冲、连接状态。
4. 再进入 HTTP 请求解析和响应构造。
```
