# 12 复习问题与行动项

## 1. 自测问题

```text
1. 单进程阻塞服务器为什么一次只能处理一个客户端？
2. 多进程服务器中父进程为什么可以 close client_fd？
3. 多线程服务器中主线程为什么不能随便 close client_fd？
4. 线程池 worker 为什么不能持有任务队列锁执行 task？
5. 线程池阻塞模型为什么会被慢客户端耗尽 worker？
6. select 为什么每轮都要 read_set = master_set？
7. select 返回 ready_count 后为什么还要遍历 fd？
8. recv 返回 0 表示什么？
9. remove_client 为什么必须 close + FD_CLR + erase + rebuild？
10. closed_fds_ 为什么用完必须 clear？
11. 旧版 handle_client 为什么不适合直接提交给 select + pool？
12. worker 在当前模型中为什么不应该 close fd？
13. worker 直接 write fd 有哪些风险？
14. poll/epoll 为什么不能自动解决 fd 生命周期问题？
15. 标准 Reactor 中 worker 应该负责什么？
```

## 2. 当前最小行动项

```text
1. 保留并复习单进程、多进程、多线程、线程池、select、select+pool 六个模型。
2. 为每个模型写一段“解决了什么 / 引入了什么问题”。
3. 修正当前 select + pool 中的职责注释。
4. 做 3 个测试：单客户端 echo、多客户端 echo、nc -N 断开检测。
5. 再进入 poll 改写。
```
